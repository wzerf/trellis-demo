### 2.2 5 个手写 warnings 根因(更新结论)

**第一轮调查(2026-06-15 凌晨)**:以为是 `@author` 中文"作"字 typo。已修(改成 `@author`)。

**第二轮调查(2026-06-15 02:00+):修了 `@author` 后 5 个 UnrecognisedJavadocTag 仍存在**。根因 = EP 2.50.0 javadoc parser bug:

看 EP 2.50.0 源码 (`core/src/main/java/com/google/errorprone/bugpatterns/javadoc/UnrecognisedJavadocTag.java`):

```java
private static final Pattern TAG = Pattern.compile("\\{@(code|link)");
// ...
int end = text.indexOf('}', start);
tags.put(comment.getSourcePos(start), tag);
```

EP 用 raw text regex 找 `{@code`/`{@link` 配 `text.indexOf('}')`,再用 `getSourcePos()` 转 source 位置。**raw byte 位置**(在含中文等多字节字符时)与 DocTree 解析位置不一致 → 误报。

最小复现: BizException line 17 的 `/** 业务码（来自 {@link ResultCode} 或自定义） */` → EP 报 "mismatched braces" 错。这 javadoc 完合法(standard inline tag,平衡的 `{}`)。EP 内部 raw-text vs DocTree 不对齐导致误报。

**结论**: 5 个 `UnrecognisedJavadocTag` 都是 EP 2.50.0 false positive,无源码可改。需接受为基线。

附带验证(2026-06-15 02:10+):

- 简化 4 个文件 javadoc(删 `<p>`/`<strong>`/`<code>`/`<ul>`/`<li>` 等 HTML 标签 + 改 `<code>{}</code>` 为 `{}`)→ warnings 不变
- EP `-Xep:UnrecognisedJavadocTag:OFF` 旗标 MCP 3.x fork 模式不传(EP 2.50.0 已知缺陷)
- EP `errorprone.xml` 配置靠 `-XepOpt:Configuration=...`,MCP 也不传
- 退路:等 EP 升 2.51 修这个 bug,或换 `<` `>` 转义(但 javadoc 风格变化)

### 2.3 2 个生成代码 warnings 根因(已验证)

`target/generated-sources/.../SysUserProxy.java:96-99` 是 Easy-Query APT 自动生成的字段 javadoc:

```java
/**
 * 数据库列的简单获取
 * @return                              // ← 空描述 + 字段上不该用 @return
 */
public SysUserProxyFetcher FETCHER = ...;  // 这是字段,不是方法
```

EP 正确抓出两个错:

- `InvalidBlockTag`: `@return` 不允许在字段上
- `EmptyBlockTag`: `@return` 描述为空

**这是 Easy-Query APT 生成器的 bug**,不是我们写的代码。Easy-Query 版本 `3.2.12`。

- 该文件有类级 javadoc(标着"this file automatically generated by easy-query")
- 但**没有 `@Generated` 标注**,所以 EP 不把它当生成代码跳过(EP 2.50.0 跳生成代码靠 `@Generated` 或 `lombok.Generated` 识别)
- 同时 EP 2.50.0 也不消费 `-XepDisableWarningsInGeneratedCode` 旗标(javac 报"无效的标记")
